# Тестирование приложения

## Назначение документа

Файл описывает порядок проверки работоспособности приложения для интеллектуальной классификации и пакетной сортировки документов, расположенного в каталоге `project/`.

## Цели тестирования

В ходе проверки подтверждаются следующие свойства системы:

- корректная классификация текста документа;
- корректная классификация скана документа;
- корректная пакетная обработка ZIP-архива документов;
- наличие и актуальность обученных моделей и отчетных артефактов;
- корректная работа REST API и локальной базы данных;
- готовность проекта к локальному запуску на другой машине.

## Состав проверок

### 1. Автоматические тесты `pytest`

Автотесты находятся в каталоге `project/tests/` и покрывают ключевые сценарии работы приложения.

Проверяемые направления:

1. Распознавание счета по текстовому описанию через сервисный слой.
2. Распознавание приказа по скану документа через сервисный слой.
3. Классификация изображения документа по файловому пути.
4. Проверка наличия артефактов обучения: моделей, диаграмм и графиков.
5. Проверка того, что реестр моделей хранит только текущие относительные пути к артефактам.
6. Пакетная обработка ZIP-архива и формирование итогового архива результата.
7. Проверка одноразовой инициализации моделей при повторном `ensure_ready()`.
8. Проверка API-метода `POST /api/text/classify`.
9. Проверка API-метода `POST /api/image/classify`.
10. Проверка API-метода `POST /api/batch/classify-archive`.
11. Проверка API-метода `GET /api/history`.
12. Проверка API-метода `GET /api/batch/history`.
13. Проверка перенаправления корневого маршрута при отключенном UI.
14. Проверка выбора основного порта, если он свободен.
15. Проверка подбора следующего свободного порта.
16. Проверка ошибки при отсутствии свободных портов в заданном диапазоне.

Основные тестовые файлы:

- `project/tests/test_service.py`
- `project/tests/test_api.py`
- `project/tests/test_main.py`
- `project/tests/conftest.py`

### 2. Smoke-проверка приложения

После запуска приложения дополнительно выполняется запрос к служебному маршруту:

- `GET /api/health`

Проверка подтверждает, что сервис поднят и обе модели готовы к работе.

### 3. Ручная функциональная проверка

Дополнительно рекомендуется открыть интерфейс Gradio:

- `http://127.0.0.1:8000/ui`

Если порт `8000` занят, приложение автоматически стартует на ближайшем свободном порту.

Минимальный набор ручных действий:

1. Ввести текст документа и получить тип документа с вероятностями.
2. Загрузить скан документа и проверить результат классификации.
3. Загрузить демонстрационный ZIP-архив и убедиться, что система возвращает итоговый архив с сортировкой по папкам.
4. Проверить отображение истории одиночных запросов и истории пакетных прогонов.
5. Проверить блок метрик обученных моделей.

## Команды запуска проверок

Все команды удобно выполнять из корня репозитория `multimodal-data-classifier/`.

### Запуск приложения

```powershell
python main.py
```

Или напрямую из каталога `project/`:

```powershell
python project/main.py
```

### Пересборка моделей и артефактов

```powershell
python generate_assets.py
```

### Запуск автотестов

```powershell
python -m pytest --cov=project/vkr_classifier --cov-report=term-missing project/tests
```

## Ограничения текущего тестирования

Автотесты не покрывают полностью визуальную часть интерфейса Gradio, поэтому итоговая приемка должна включать короткую ручную проверку пользовательского интерфейса.

Кроме того, качество моделей подтверждено на синтетическом наборе данных. Для перехода к промышленному использованию потребуется проверка на реальном корпусе документов.
